## 4. 每一项应该准备成什么样

### 4.1 源码或网络链接

建议提交内容：

- 项目仓库链接，或源码压缩包
- 附 `README.md`
- 附简单运行说明

提交前检查：

- [ ] 删除或排除真实配置文件
- [ ] 删除或排除 `.env`
- [ ] 检查压缩包内是否有无关临时文件
- [ ] 确认项目名称统一

建议说明语句：

> 本作品为基于树莓派的文明用语语音智能体系统，采用“语音输入 + 大模型判断 + 屏幕与语音反馈”的工作流实现。

### 4.2 项目基本信息表

至少需要填写：

- 作品名称
- 成果附件清单
- 辅导老师信息
- 300 字以内作品简介
- 参赛成员与分工

建议先单独准备以下内容再填写：

- 作品中文名
- 100 字短简介
- 300 字正式简介
- 个人分工说明

### 4.3 项目报告 PPT

建议最少包含以下页面：

1. 封面
2. 项目背景和问题
3. 作品目标
4. 硬件组成
5. 工作流设计图
6. 系统模块说明
7. 关键代码或关键功能
8. 测试过程
9. 作品效果展示
10. 遇到的问题与改进

小学组建议：

- 每页字不要太多
- 多放流程图、照片、界面截图
- 讲清楚“输入是什么、判断是什么、输出是什么”

### 4.4 报告视频

视频中建议包含以下片段：

1. 身份认证
2. 作品介绍
3. 树莓派设备展示
4. 实际运行演示
5. 工作流讲解
6. 开发过程照片或录屏
7. 总结

视频要求检查：

- [ ] 格式为 `MP4`
- [ ] 分辨率不低于 `1280x720`
- [ ] 文件小于 `200MB`
- [ ] 人声清楚
- [ ] 画面稳定

## 5. 这个项目如何对应“小学组工作流”

为了方便答辩和写 PPT，建议把本项目统一表述成下面这个版本：

| 节点 | 节点名称 | 作用 |
| --- | --- | --- |
| 1 | 唤醒词检测 | 等待用户叫醒机器人 |
| 2 | 语音活动检测 | 判断用户是否开始说话 |
| 3 | 录音 | 录下用户说的话 |
| 4 | 语音转文字 | 把语音转成文字 |
| 5 | 大模型文明判断 | 判断说话是否文明 |
| 6 | 屏幕显示结果 | 显示评分和建议 |
| 7 | 语音播报建议 | 用语音提醒更文明的说法 |

这样写的好处：

- 满足小学组“5 个以上节点”的要求
- 有明确的大模型判断节点
- 容易画流程图
- 容易讲清楚输入、处理、输出
