## 2. 技术路线总览

这一部分回答“整个项目是按什么路线做出来的”。

建议先写总路线，再写分路线。

### 2.1 总体技术思路

建议写法：

本项目采用“树莓派本地设备 + 云端大模型能力 + 本地交互输出”的技术路线。

其中：

- 树莓派负责设备运行、语音采集、界面显示和音频播放
- 云端语音识别负责把语音转换成文字
- 云端大模型负责分析文明程度并生成建议
- 本地显示和语音模块负责把结果反馈给用户

### 2.2 技术路线一句话总结

建议格式：

> 先在树莓派上接收语音输入，再调用云端 AI 完成识别和判断，最后把结果通过屏幕和语音反馈给用户，形成一个完整的智能体工作流。

### 2.3 技术路线分层

建议从下到上分四层写：

1. 硬件层
2. 设备能力层
3. AI 能力层
4. 交互反馈层

可以用下面这个结构描述：

```text
硬件层：树莓派、麦克风、喇叭、显示屏
设备能力层：唤醒、录音、播放、显示
AI 能力层：语音转文字、大模型文明判断
交互反馈层：结果展示、语音提醒、返回待机
```

### 2.4 为什么选择这条技术路线

这里建议写“选择理由”，后面 PPT 很好用：

- 树莓派适合做软硬结合的智能终端

- 模块化结构便于后续继续扩展

## 3. 工作流总览

这一部分回答“它到底是怎么一步一步工作的”。

这是比赛里最关键的一部分，后面 PPT 和答辩都要重点使用。

### 3.1 工作流定义

建议写法：

本项目采用基于工作流的 AI 智能体设计。系统不是只执行单一程序，而是按照固定节点顺序完成感知、处理、判断、反馈，再回到待机状态。

### 3.2 工作流主流程

当前建议统一使用这一版：

```text
唤醒词检测
-> 等待用户说话
-> 录音
-> 语音转文字
-> 大模型判断文明程度
-> 屏幕显示结果
-> 语音播报建议
-> 返回待机
```

初稿可直接写法：

本项目的主工作流从“唤醒词检测”开始。机器人平时处于待机状态，持续监听用户是否说出目标唤醒词“饭团饭团”。当系统检测到唤醒词后，会先进入等待用户继续说话的阶段，确认后续确实存在语音输入，再启动录音模块采集用户语音。

录音完成后，系统会把这段语音交给语音转文字模块处理，将用户说的话转换为中文文本。随后，程序把这段文本送入大模型文明判断模块，由模型分析表达是否文明，并返回文明评分、原因说明和更合适的表达建议。

在得到分析结果后，系统会把结果显示在屏幕上，并在需要时通过语音播报提醒用户。如果本轮交互结束，系统会自动回到待机状态，继续等待下一次唤醒。整个流程形成了“感知、处理、判断、反馈、返回待机”的完整闭环。

### 3.3 小学组可用的节点拆分

为了更符合比赛表达，建议把工作流固定写成 7 个节点：

| 节点序号 | 节点名称 | 节点作用 |
| --- | --- | --- |
| 1 | 唤醒词检测 | 让机器人知道用户在叫它 |
| 2 | 等待用户说话 | 判断用户是否开始输入语音 |
| 3 | 录音 | 采集用户语音 |
| 4 | 语音转文字 | 把录到的语音变成文字 |
| 5 | 大模型文明判断 | 判断说话是否文明，并生成建议 |
| 6 | 屏幕显示结果 | 让用户看到评分和建议 |
| 7 | 语音播报建议 | 让机器人把建议说出来 |

### 3.4 工作流中的“智能判断节点”

这部分一定要明确写出来，因为比赛特别看重这个点。

建议固定表述：

本项目的智能判断节点是“`大模型文明判断`”。

这个节点不是简单执行固定命令，而是根据用户说话内容进行分析，输出：

- 是否文明
- 文明评分
- 原因说明
- 更合适的表达建议

初稿可直接写法：

本项目最核心的智能判断节点是“大模型文明判断”。这个节点和普通程序中“遇到某个词就执行某个固定命令”的逻辑不同，它不是简单匹配预设答案，而是要结合用户说话的具体内容进行理解和判断。

在实际运行中，系统会先把用户语音转换成文字，再把这段文字发送给大模型。模型需要根据提示词判断这句话是否文明，并以结构化 JSON 的形式返回结果，包括是否文明、文明评分、原因说明以及更礼貌的表达建议。这样做的好处是，系统不仅能识别明显的脏话，也能识别语气生硬、嘲讽、轻度攻击等“不够文明”的表达。

因此，这个节点体现了本项目真正的“智能体判断能力”。它不是只执行固定代码，而是在工作流中承担了语义理解、文明评估和建议生成三个任务，也是最能体现 AI 技术价值的部分。

### 3.5 工作流输入、处理、输出

建议统一写法：

- 输入：用户说的话
- 处理：语音识别 + 大模型判断
- 输出：屏幕结果 + 语音提醒

### 5.1 唤醒词检测模块

建议写的点：

- 机器人平时处于待机状态
- 用户说出唤醒词后，系统进入下一步
- 这样可以减少误触发，也更像真实机器人交互

建议重点补充的技术细节：

- 当前项目的长期待机唤醒优先采用本地离线方案，不依赖云端大模型
- 唤醒引擎使用 `openWakeWord`，目标唤醒词为“饭团饭团”
- 麦克风持续采集音频后，会先做采样率匹配，再送入唤醒模型判断
- 为了降低误触发，系统可以结合阈值、连续命中帧数、静音间隔等条件进行过滤
- 唤醒成功后，系统不会立刻进入大模型判断，而是先进入“等待用户继续说话”和“录音”阶段

这一部分如果要写进正式报告，最适合放在：

可补充代码位置：

- `modules/wakeword/`
- `modules/wakeword/openwakeword_wakeword.py`
- `modules/wakeword/vosk_wakeword.py`

初稿可直接写法：

本项目的唤醒词检测模块负责让机器人在长期待机状态下持续监听用户是否在呼叫它。当前项目优先采用本地离线唤醒方案，不依赖云端大模型服务，从而降低待机成本并提升响应稳定性。

在具体实现上，系统使用 `openWakeWord` 作为主要唤醒引擎，目标唤醒词设置为“饭团饭团”。麦克风会持续采集音频，程序先对输入音频进行采样率匹配，再送入唤醒模型判断是否命中目标词。为了减少误触发，系统还可以结合阈值、连续命中帧数和唤醒间隔等条件进行过滤。

当系统识别到唤醒词后，不会立刻进入大模型分析，而是先切换到“等待用户继续说话”的状态，然后开始录音并进入后续处理流程。这样的设计既保留了真实机器人的交互感，也能避免无意义的云端调用。

### 5.4 大模型文明判断模块

这是全文最关键的技术点之一，建议重点展开。

建议写的点：

- 大模型读取语音识别后的文字
- 判断表达是否文明
- 输出结构化结果
- 给出更礼貌的表达建议

建议固定强调输出内容：

- `civilized`
- `score`
- `reason`
- `suggestion`

建议重点补充的技术细节：

- 大模型收到的输入不是原始音频，而是“语音转文字”后的中文文本
- 系统通过固定提示词要求模型只返回 JSON，避免返回大段自然语言造成解析困难
- 当前评分采用 `0-100` 的文明分数，其中高分表示更文明
- 提示词中明确了评分档位、`civilized` 与分数的对应关系，以及改写建议的生成要求
- 程序会先校验模型返回的 JSON 字段是否完整，再把结果转换成结构化对象
- 在当前运行逻辑中，只有当 `civilized` 为 `false` 时，系统才会进入提醒播报；如果已经是文明表达，则直接正向提示或返回待机

这一部分如果要写进正式报告，最适合放在：

- 第 3.4 节“工作流中的智能判断节点”：强调这一步是项目最核心的 AI 判断环节

- 第 6.1 节“数据流”：说明文本如何进入模型，再变成结构化判断结果
- 第 9.1 节“功能验证”：说明如何验证模型能正确识别不文明表达并给出建议

可补充代码位置：

- `modules/llm/openai_civil_analyzer.py`
- `modules/llm/deepseek_civil_analyzer.py`
- `modules/models.py`

初稿可直接写法：

大模型文明判断模块是本项目最核心的智能判断节点。它的作用不是处理原始音频，而是读取“语音转文字”之后得到的中文文本，再对这句话是否文明进行判断，并生成结构化结果。

当前系统调用云端大模型完成这一任务。程序会向模型发送固定提示词，要求模型严格按照 JSON 格式返回四个字段：`civilized`、`score`、`reason` 和 `suggestion`。其中，`civilized` 表示表达是否文明，`score` 表示文明程度评分，`reason` 表示评分原因，`suggestion` 表示更礼貌的改写建议。

为了让评分更稳定，提示词中已经加入了明确的分档规则。例如，90 到 100 分表示“非常文明”，70 到 89 分表示“基本文明”，40 到 69 分表示“不够文明”，0 到 39 分表示“明显不文明”。如果出现脏话、辱骂、人身攻击等表达，模型会优先判定为不文明，并给出较低分数和替代表达建议。

程序在收到模型返回结果后，会先校验 JSON 是否完整，再把结果转换为结构化对象，供后续显示和播报模块使用。在当前运行逻辑中，如果表达已经文明，系统可以直接给出正向提示或返回待机；如果表达不文明，系统则会显示分析结果，并通过语音播报提醒用户换一种更礼貌的说法。
